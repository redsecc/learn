## Objective

Logged in as your own user, access a resource that belongs to a *different*
user by tampering with an object reference in the request. This lab uses
**OWASP Juice Shop**, a deliberately insecure application maintained for
training.

## Environment setup

Run Juice Shop locally — no account or remote target required:

```bash title="Run Juice Shop in Docker"
docker run --rm -p 127.0.0.1:3000:3000 bkimminich/juice-shop
```

Then browse to `http://localhost:3000`, register a throwaway account, and log
in. Proxy your traffic through Burp Suite or use your browser's dev tools to
inspect and replay requests.

<Callout type="note" title="Why a local target">
  RedSec is static and hosts no infrastructure. You run the vulnerable app
  yourself, on your own machine, so you're always testing something you own.
</Callout>

## Your task

1. Find a request that fetches an object tied to *your* account (a basket, a
   review, an order).
2. Identify the object reference in that request.
3. Change it to reference an object you don't own.
4. Confirm you can read or modify it.

## Hints

<details>
  <summary>Hint 1 — what to watch</summary>

  Browse the app with dev tools open on the Network tab. Look for API calls like
  <code>/rest/basket/&lt;id&gt;</code> that include a small numeric identifier.
</details>

<details>
  <summary>Hint 2 — the probe</summary>

  Note the ID the app uses for *your* basket, then replay the same request — with
  your own session cookie — using a neighbouring ID (try `1`, `2`, …). Does the
  server return data that isn't yours?
</details>

<details>
  <summary>Hint 3 — try other methods</summary>

  Authorization is sometimes enforced on <code>GET</code> but not on
  <code>PUT</code>/<code>POST</code>. If reading is blocked, try modifying.
</details>

---

## Solution

<details>
  <summary>Reveal the full solution</summary>

  After logging in, add an item to your basket and watch the request:

  ```http
  GET /rest/basket/6 HTTP/1.1
  Host: localhost:3000
  Authorization: Bearer <your-token>
  ```

  `6` is *your* basket id. Replay the request with a different id while keeping
  your own token:

  ```http
  GET /rest/basket/1 HTTP/1.1
  Host: localhost:3000
  Authorization: Bearer <your-token>
  ```

  The server returns basket `1` — which belongs to another user. The endpoint
  trusts the id in the URL and never checks that the basket belongs to the
  authenticated user. That's an IDOR.

  **The fix:** scope every lookup to the authenticated principal server-side,
  e.g. `WHERE basket.id = :id AND basket.user_id = :current_user`.
</details>

<Callout type="warning" title="Stay in scope">
  Only run this against your own local Juice Shop instance. Never apply IDOR
  probing to applications you aren't authorized to test.
</Callout>
