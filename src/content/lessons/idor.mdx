**IDOR** (Insecure Direct Object Reference) is an access-control flaw: the
application exposes a reference to an internal object — a database row, a file, a
user — and fails to verify that the requester is actually allowed to access it.
It is one of the most common and impactful web vulnerabilities precisely because
it's so easy to introduce.

## Spot the pattern

Anywhere an identifier appears in a request, ask "what happens if I change it?"

```http title="The request"
GET /api/invoices/1043 HTTP/1.1
Host: app.example.com
Cookie: session=<your-session>
```

```http title="The probe — same session, a different ID"
GET /api/invoices/1042 HTTP/1.1
Host: app.example.com
Cookie: session=<your-session>
```

If the second request returns *someone else's* invoice, the endpoint trusts the
ID without an ownership check. That's an IDOR.

## Where to look

- Numeric IDs in paths or query strings (`?id=`, `/users/42`).
- IDs in JSON bodies, including nested ones.
- Filenames in download/export endpoints.
- IDs in hidden form fields and API responses you can replay.

<Callout type="tip" title="Don't trust opaque IDs">
  UUIDs and hashes feel safe but only add obscurity, not authorization. If you
  can obtain another user's identifier (from a share link, an email, a
  different endpoint), an unprotected object reference is still exploitable.
</Callout>

## Test methodically

Use two accounts you control. Capture a request as user A, then replay it with
user B's session against user A's object IDs. Vary the **method** too — an
endpoint might enforce authorization on `GET` but not `DELETE` or `PUT`.

```http title="Try other verbs"
DELETE /api/invoices/1042 HTTP/1.1
Host: app.example.com
Cookie: session=<user-B-session>
```

<Callout type="warning" title="Handle data carefully">
  Reading or modifying other users' records — even to prove a bug — can cause
  real harm. Stay within the rules of engagement, touch the minimum needed to
  demonstrate impact, and never exfiltrate real personal data.
</Callout>

## The fix (so you can explain it)

Every request must enforce authorization *server-side*, scoping objects to the
authenticated principal:

```sql title="Authorize at the data layer"
SELECT * FROM invoices WHERE id = ? AND owner_id = :current_user;
```

## Recap

- IDOR = object reference + missing ownership check.
- Probe every identifier, in every parameter and every HTTP method.
- The fix is server-side, per-request authorization scoped to the user.
