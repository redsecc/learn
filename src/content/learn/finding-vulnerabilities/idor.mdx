**IDOR** (Insecure Direct Object Reference) is an access-control flaw: the app
exposes a reference to an object — a row, a file, a user — and fails to verify
the requester is allowed to access it.

## Spot the pattern

Anywhere an identifier appears in a request, ask "what if I change it?"

```http title="The request"
GET /api/invoices/1043 HTTP/1.1
Cookie: session=<your-session>
```

```http title="The probe — same session, a different ID"
GET /api/invoices/1042 HTTP/1.1
Cookie: session=<your-session>
```

If the second request returns *someone else's* invoice, the endpoint trusts the
ID without an ownership check.

## Where to look

- Numeric IDs in paths/queries (`?id=`, `/users/42`).
- IDs in JSON bodies, including nested ones.
- Filenames in download/export endpoints.
- IDs in hidden fields and API responses you can replay.

<Callout type="tip" title="Opaque IDs aren't authorization">
  UUIDs only add obscurity. If you can obtain another user's identifier, an
  unprotected reference is still exploitable. Also vary the HTTP method — auth is
  often enforced on <code>GET</code> but not <code>PUT</code>/<code>DELETE</code>.
</Callout>

## The fix

Authorize every request server-side, scoped to the authenticated user:

```sql title="Authorize at the data layer"
SELECT * FROM invoices WHERE id = ? AND owner_id = :current_user;
```

<Callout type="warning" title="Handle data carefully">
  Touch the minimum needed to prove the bug, and never exfiltrate real personal
  data.
</Callout>
